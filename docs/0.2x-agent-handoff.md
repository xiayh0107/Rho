# Rho 0.2.x Agent Handoff Plan

> Archived handoff: this document describes the former desktop-first workbench
> line. ADR-010 supersedes that product boundary; retain this file only for
> implementation history and migration context.

Date: 2026-07-16  
Baseline: `0.2.0-dev.1`

## How To Use This Brief

This document is an implementation handoff, not a request to redesign the
architecture. The implementation agent should complete one work package at a
time, run the listed gates, and report changed files and residual risks. A
separate review agent will then inspect the diff and rerun the gates.

Before implementation or packaging, read `docs/windows-build-environment.md`.
It records the current machine's verified toolchain, release target, build
commands, artifact checks and required handoff report.

The implementation agent must preserve:

- Workspace R as the only authority for live R objects and execution;
- Agent R as the `aisdk` host, never as a second scientific workspace;
- Rust broker ownership of Ark transport, revisions, approvals and SQLite;
- no Python, Jupyter Server, JupyterLab or Electron dependency;
- existing draggable panel behavior and browser-mode frontend fallback;
- existing aisdk family repositories, unless a concrete upstream gap is
  demonstrated and separately approved.

## Current Baseline

Already available in `0.2.0-dev.1`:

- Tauri Windows prototype and NSIS packaging;
- Ark-backed persistent Workspace R;
- R Console, Plots, Problems, Environment and Agent panels;
- draggable/persistent panel dividers;
- project root and source-file listing API;
- multi-document frontend state;
- project file read, save and create commands;
- Workspace R `setwd()` synchronization;
- path traversal protection and source-extension allowlist;
- Rust workspace tests and project path tests.

The remaining work should build on these APIs rather than replacing them.

## Work Packages

### WP1: Project Opening And Session Restoration

Goal: make project selection a real daily workflow.

Implement:

- native Windows directory picker through a narrow Tauri command;
- project metadata under the app-local data directory, keyed by normalized
  project root;
- last-opened project restoration at startup;
- current document, open document list, cursor positions and panel sizes
  restored per project;
- explicit `project unavailable` state when a saved root no longer exists;
- refresh file tree after external file changes, with no silent overwrite of a
  dirty document.

Preferred files:

- `desktop/src-tauri/src/main.rs` or a new small `project.rs` module;
- `desktop/dist/app.js`, `index.html`, `styles.css`;
- `docs/windows-prototype.md` and tests.

Acceptance tests:

- choose a directory, restart Rho, and see the same project and document;
- close/reopen a document with unsaved text and receive an explicit choice;
- a path outside the project root is rejected;
- an external edit is detected and does not overwrite dirty editor content;
- project state survives a normal application restart.

Do not implement cloud sync, multi-user locking or a general file manager.

### WP2: Editor And Source Execution

Goal: replace the textarea prototype with a real source-editing surface.

Implement:

- language-aware R editor with syntax highlighting and matching brackets;
- source selection execution and current-line execution;
- source-file execution with visible origin and file path;
- completion only if it can be kept bounded and local; do not block the
  project workflow on a full language server integration;
- dirty state, save state and cursor position tied to the document model.

Preferred approach:

- use the lightest editor that fits the existing Tauri/browser frontend;
- if Monaco is selected, load it in a way that does not require a runtime
  Node process in the installed application;
- keep the Workbench Protocol independent of editor internals.

Acceptance tests:

- execute a selection and the current line without executing unrelated code;
- execute a complete `.R` file and link the run to its path and revision;
- syntax/selection state remains correct after panel resizing;
- save/reopen preserves content and cursor position.

Do not add notebook semantics, Python syntax or an xterm-based R Console.

### WP3: Structured Runs, Problems And Recovery

Goal: make ordinary R errors recoverable and auditable.

Implement:

- durable run records for user, agent and system executions;
- explicit run states: queued, running, waiting, completed, failed,
  cancelled, interrupted and crashed;
- structured Problems entries containing message, call, traceback, source path,
  execution id and workspace identity;
- Retry, Explain and Open Source actions;
- cancellation command with cooperative cancel followed by bounded Ark
  interrupt;
- Workspace R restart detection and user-visible recovery state;
- Agent R restart without losing Workspace R or prior run records.

Acceptance tests:

- an R error appears as a Problems record, not only Console text;
- Retry creates a new linked run and does not mutate the original record;
- cancellation returns Ark to idle and a follow-up expression succeeds;
- a broker restart marks incomplete runs as interrupted;
- an Agent crash can be restarted while Workspace R retains its object.

Do not introduce a second event database or parse terminal output for errors.

### WP4: Approval And Agent Continuation UX

Goal: make Act mode reviewable instead of implicitly trusted.

Implement:

- broker approval request/response state surfaced in the Agent timeline;
- approve, reject and cancel controls with request id and exact tool/code;
- clear mode policy: Ask read-only, Plan read-only plus proposal, Act mutable
  only after approval;
- continuation after rejection or stale revision;
- Agent turn history persisted independently from Workspace R;
- tool results show code, duration, success/error and revision transition.

Acceptance tests:

- Ask cannot execute `run_r`;
- Plan cannot execute mutation before confirmation;
- Act displays exact R code before approval;
- rejection leaves Workspace R unchanged;
- stale revision causes refresh/replan rather than silent execution;
- a completed Agent turn remains visible after Agent R restart.

Do not add unrestricted shell access, MCP child processes or autonomous package
installation in this work package.

### WP5: Project-Scoped Scientific Environment Basics

Goal: expose enough environment state for reproducible local work.

Implement:

- project `renv` detection and read-only status first;
- R version, library paths, Bioconductor version and loaded package summary;
- bounded data-frame viewer and export of visible summaries;
- plot history with run/source provenance;
- structured Quarto/R Markdown render errors, without making Quarto a required
  startup dependency.

Acceptance tests:

- environment drift is visible before a run;
- large objects are summarized without full serialization;
- a plot links back to source file, execution id and revision;
- a second user can identify the environment needed to reproduce a selected
  result.

Do not implement remote compute, cloud storage or broad semantic adapters until
the local environment contract is tested.

## Recommended Sequence

1. WP1 project opening and restoration.
2. WP2 editor/source execution.
3. WP3 structured runs, Problems and recovery.
4. WP4 approval and Agent continuation.
5. WP5 scientific environment basics.

WP1 and WP2 may share frontend work, but WP3 must not be postponed until after
adding more Agent features. A work package is complete only when its acceptance
tests pass and its failure behavior is documented.

## Review Protocol

When implementation is handed back for review, report:

- commit or working-tree diff identifier;
- changed files grouped by work package;
- commands and exact test counts;
- manual workflow tested;
- known limitations and follow-up items;
- whether any aisdk family change was made and why.

The review agent should inspect findings first, in this order:

1. Workspace R authority and stale revision correctness;
2. data loss or dirty-document overwrite;
3. cancellation/crash/restart behavior;
4. approval policy bypass;
5. path traversal, credential leakage and event persistence;
6. UI regressions and cross-platform assumptions;
7. missing tests and documentation drift.

## Definition Of 0.2.x Done

The milestone is complete only when a user can:

1. open a local R project;
2. edit and save multiple R files;
3. execute a selection or file in the same persistent Workspace R;
4. inspect an object and plot with provenance;
5. encounter an error and use Problems to explain, fix or retry;
6. ask the Agent, review an approval, and see the resulting revision;
7. cancel or recover a run without losing project or Workspace R state;
8. restart Rho and recover project files, run history and Agent timeline.
